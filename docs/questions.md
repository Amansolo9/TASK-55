# ClubOps Governance & Finance Portal - Questions & Solutions

1. Offline Multi-Site Data Synchronization
Question: The prompt specifies "offline, multi-site" operation with SQLite as the sole data store. SQLite is a single-file embedded database. How do multiple physical campuses share member records, budgets, and audit logs if each site operates offline with its own SQLite instance?
My Understanding: True offline operation means each site must function independently without network connectivity. Yet organizers and administrators need a unified view across all sites for budgets, members, and reporting.
Solution: Each site runs its own SQLite instance as the source of truth for local operations. A manual "sync bundle" export/import mechanism (signed JSON or encrypted SQLite WAL snapshots) allows a designated administrator to carry data between sites via USB or local network transfer when connectivity is available. Conflict resolution uses a last-write-wins policy keyed on entity ID + updated_at timestamp, with conflicts logged to the audit trail for manual review. The `campus_code` field on budgets and `site_id` on fulfilled orders already partition data by location, making merge boundaries clear.

2. Administrator Bootstrap and Lockout Recovery
Question: Authentication is local-only with no email, SMS, or third-party recovery. Accounts lock after 5 failed attempts for 15 minutes. What happens if the sole administrator is locked out, or forgets their password with no other admin to initiate a reset?
My Understanding: Every access-recovery path depends on an existing administrator. A single point of failure exists if there is only one admin account.
Solution: The application seeds a bootstrap admin account on first startup using the `APP_BOOTSTRAP_ADMIN_PASSWORD` environment variable. If the sole admin is locked out, an operator with server/container access can restart the application with a new bootstrap password, which re-seeds or resets the admin credentials. This keeps recovery offline and within the server operator's control without exposing any external recovery surface. The lockout timer (15 minutes) is short enough that temporary lockouts self-resolve, but catastrophic password loss requires server-level intervention, which is documented in the deployment runbook.

3. CSV Import Atomicity and Partial Failure Handling
Question: Bulk CSV import supports up to 5,000 rows with "inline validation and a downloadable error report." Should the import be all-or-nothing (atomic), or should valid rows import while invalid rows are rejected?
My Understanding: In an offline environment where re-importing is easy, partial imports create confusion about which rows actually persisted. However, rejecting an entire 5,000-row file for one bad row is operationally painful.
Solution: The import uses a two-phase approach. Phase 1 validates all rows and returns an error report CSV listing every row with issues (row number, field, reason). No rows are persisted during validation. Phase 2 only executes if the user explicitly confirms after reviewing the report, at which point all valid rows are inserted in a single SQLite transaction. If any database-level error occurs during Phase 2 (e.g., uniqueness violation), the entire transaction rolls back. This gives operators a clean preview before committing, and the downloadable error report lets them fix and re-upload rather than guess which rows failed.

4. Budget Period Rollover and Spent-Amount Reset
Question: Budgets are set as "monthly or quarterly" with real-time execution rates and projected end-of-period balances. When a period ends (e.g., March to April for monthly), does the spent amount reset? Is remaining budget carried forward? What triggers the transition?
My Understanding: The `budgets` table stores `period_start` and `amount` and `spent`, but there is no automated period-rollover mechanism. Without explicit rollover logic, stale budgets accumulate and threshold alerts become meaningless.
Solution: A background worker (similar to the existing threshold-alert worker) runs daily and checks if any budget's period has elapsed. When a period ends, the system automatically creates a new budget row for the next period with `spent = 0`, copying the previous period's `amount` as the default. The old period's row is marked `status = 'closed'` and becomes read-only for reporting. Carryover is not automatic; any remaining balance stays on the closed record for audit purposes. If an organizer wants to adjust the new period's amount, the standard change-request workflow applies (auto-approved if <=10%, approval required if >10%).

5. Credit Rule Effective-Date Overlap and Gap Handling
Question: The credit rule engine uses versioned rules with `effective_from` and `effective_to` date ranges. What happens if an administrator creates two rule versions with overlapping date ranges, or if there is a gap between versions where no rule is active?
My Understanding: The prompt states "calculations always use the rule version active on the transaction date." If zero or multiple rules match a given date, the calculation has no deterministic answer.
Solution: On rule creation, the service validates that the new version's date range does not overlap with any existing active version. If it does, the request is rejected with a specific error identifying the conflicting version. For gaps, when a credit issuance is attempted and no active rule covers the transaction date, the service returns a clear error rather than silently skipping or using a default. Administrators are shown a "coverage timeline" view on the credit rules management page that visually highlights uncovered date ranges, making gaps obvious before they cause issuance failures.

6. Review Appeal Window Enforcement
Question: Reviewers can appeal a hidden review within 7 days. Is this 7 calendar days from the review creation date, or 7 days from when the review was hidden? And should the backend enforce this deadline, or is it a UI-only constraint?
My Understanding: The current implementation enforces the 7-day window at the UI level only (hiding the appeal button), but a determined user could still submit an appeal via a direct API call after the window closes. For an auditable governance system, the backend must be authoritative.
Solution: The 7-day appeal window starts from the `hidden_at` timestamp (the moment the review was hidden by a moderator, not the review creation date), since a reviewer cannot appeal a review that has not been hidden yet. The backend enforces this by comparing `hidden_at + 7 days` against the current server time when processing an appeal request. If the window has passed, the API returns a 410 Gone response. The UI still hides the appeal button after 7 days for a clean user experience, but the backend is the authoritative gate.

7. Concurrent Budget Spend Recording (SQLite Write Contention)
Question: Multiple users may record spend against the same budget simultaneously. SQLite uses file-level locking, and the `RecordSpend` operation reads the current `spent` value and writes an updated total. How are race conditions on the `spent` field prevented?
My Understanding: Two concurrent requests could both read `spent = 100`, each add `50`, and both write `spent = 150` instead of the correct `spent = 200`. SQLite's default transaction isolation does not prevent this without explicit handling.
Solution: The `RecordSpend` operation uses a single atomic SQL statement: `UPDATE budgets SET spent = spent + ? WHERE id = ?` rather than a read-modify-write cycle. This is inherently safe under SQLite's serialized write model because the update is evaluated atomically within SQLite's write lock. The threshold-alert worker then picks up the new total on its next 60-second sweep. For additional safety, the service wraps spend recording in an `IMMEDIATE` transaction to fail fast on contention rather than silently retrying.

8. Encryption Key Rotation for Application-Layer PII
Question: Member PII (email, phone, custom fields) is encrypted at the application layer using AES-GCM with a key derived from the `APP_ENCRYPTION_KEY` environment variable. What happens when the encryption key needs to be rotated (e.g., suspected compromise, personnel change, compliance policy)?
My Understanding: Changing the environment variable immediately renders all existing encrypted data unreadable. There is no built-in key-rotation or re-encryption mechanism in the current `CryptoService`.
Solution: Implement a key-rotation utility that accepts both the old and new key as arguments. The utility reads every encrypted field (identified by the `enc:v1:` prefix), decrypts with the old key, re-encrypts with the new key under a new version prefix (`enc:v2:`), and updates the row in a transaction. The `CryptoService` is extended to attempt decryption with the current key first; if that fails and the prefix indicates an older version, it falls back to a configured previous key. Once all records are migrated, the old key is removed from config. The rotation event is logged in the audit trail (without logging any plaintext values).

9. Audit Log Storage Growth and SQLite Performance
Question: The append-only audit log has a 2-year retention policy, and every POST/PUT/DELETE mutation generates a log entry with before/after JSON state snapshots. In an active multi-site deployment, this table could grow to millions of rows. How does this affect SQLite query performance and database file size?
My Understanding: SQLite handles reads well at scale but write-heavy append-only tables with large JSON blobs can degrade vacuum and backup performance. The 2-year cleanup worker deletes expired rows, but SQLite does not reclaim disk space from deletions without VACUUM.
Solution: The audit log table is indexed on `created_at` and `entity` for query performance. The daily cleanup worker, after deleting expired rows, runs an incremental `PRAGMA incremental_vacuum` (not a full VACUUM, which locks the database) to gradually reclaim space. For deployments exceeding 1 million audit rows, the system supports an "archive export" that dumps older-than-6-month logs to a compressed JSON-lines file on local disk before deletion, preserving the 2-year record outside SQLite while keeping the active database performant.

10. Feature Flag Rollout Percentage Determinism Across Restarts
Question: Canary releases use FNV-32a hash of the username to determine if a user falls within the rollout percentage. Is this deterministic across application restarts and consistent if the rollout percentage is gradually increased?
My Understanding: The hash-based approach is deterministic for a given username and percentage, which is correct. But if a flag is set to 10%, then increased to 20%, do the original 10% of users remain in the cohort, or could some be shuffled out?
Solution: The FNV-32a hash produces a stable integer per username. The evaluation computes `hash(username) % 100` and checks if the result is less than `rollout_pct`. Because the hash is stable, increasing the percentage from 10% to 20% adds users whose hash falls in the 10-19 range while retaining all users in the 0-9 range. No user who was previously enabled gets disabled by a percentage increase. This monotonic inclusion property is inherent to the modulo-comparison approach already implemented. The only scenario where a user loses access is if the percentage is explicitly decreased, which is an intentional rollback and is logged in the audit trail.

11. RBAC Scope When a Team Lead Is Reassigned Between Clubs
Question: Team leads are scoped to a single `club_id`. If an administrator reassigns a team lead from Club A to Club B, what happens to their in-flight work: open budget change requests they submitted for Club A, reviews they moderated, and their session?
My Understanding: The middleware scopes all queries to the user's current `club_id`. After reassignment, the team lead immediately loses visibility into Club A's data, but their historical actions (audit log entries, submitted requests) still reference Club A.
Solution: On club reassignment, the system: (1) invalidates all active sessions for the reassigned user, forcing re-authentication so the new scope takes effect immediately; (2) any pending budget change requests they submitted are flagged as "submitter-reassigned" for an admin or organizer to re-review or cancel; (3) historical audit log entries remain unchanged since they are append-only and reflect the state at the time of the action. The reassignment itself is logged as an audit event with before/after club_id values.

12. Image Storage Cleanup for Deleted or Hidden Reviews
Question: Review images are stored locally in `./static/uploads/` with nanosecond-prefixed filenames. If a review is hidden by a moderator or the review data is eventually purged, what happens to the orphaned image files on disk?
My Understanding: The database tracks `image_paths` as a JSON array on the review record, but there is no garbage-collection process for files that are no longer referenced by any active review.
Solution: A weekly background worker scans the `./static/uploads/` directory and cross-references each file against `image_paths` in the `reviews` table. Files not referenced by any review are moved to a `./static/uploads/orphaned/` quarantine directory with a 30-day grace period before permanent deletion. Hidden reviews retain their image references (so images are not orphaned while an appeal is possible). Only after the appeal window expires and the review is confirmed hidden do the images become eligible for cleanup. Disk-usage metrics are exposed on the admin dashboard.

13. Referential Integrity of Sales Facts Across MDM Dimension Versions
Question: MDM dimensions are versioned, so importing a new product dimension version creates new rows. Sales fact records reference dimension codes (product_code, customer_code, etc.) directly. If a dimension code is removed or changed in a new version, do existing sales facts become orphaned?
My Understanding: The `sales_facts` table stores raw codes without a foreign key to a specific dimension version. This means facts are not tied to any particular version and code lookups could return different values depending on which version is queried.
Solution: Sales fact records are immutable historical data and always store the code as it existed at transaction time. Dimension codes are never deleted from previous versions (versions are append-only snapshots). When querying facts for reporting, the system joins against the dimension version that was active at the `transaction_date` of each fact row. Administrators are warned during dimension import if any existing codes are absent from the new version, and those codes are flagged as "deprecated" rather than removed. This ensures backward compatibility while allowing the dimension catalog to evolve.

14. HTMX Partial Responses and CSRF Token Lifecycle
Question: The UI uses HTMX to swap partial HTML fragments into the page without full reloads. The CSRF token is set as a cookie on first request. If a user's session expires during an HTMX partial interaction (e.g., they step away for 31 minutes), what does the user experience?
My Understanding: An expired session causes the auth middleware to return a redirect to `/login`. But HTMX will inject that redirect response (or a 401/403) into the target div rather than navigating the full page to the login screen.
Solution: The auth middleware detects HTMX requests via the `HX-Request` header. When a session has expired on an HTMX request, the server returns a 401 status with the `HX-Redirect: /login` response header instead of a standard 302 redirect. HTMX natively handles `HX-Redirect` by performing a full-page navigation to the login URL, giving the user a clean login experience rather than a broken partial swap. The CSRF token cookie survives independently of the session, so it does not need to be reissued on re-authentication.

15. Password Rotation Enforcement for Non-Administrator Roles
Question: The prompt states "administrators must rotate passwords every 180 days." Does this policy apply only to administrators, or to all roles? And what is the enforcement mechanism if an admin ignores the rotation prompt?
My Understanding: The `password_set_at` field on the users table tracks when the password was last changed, and the `must_change_password` flag forces a change at next login. But the prompt specifically calls out administrators only.
Solution: The 180-day rotation policy applies only to users with the `admin` role, as they have the highest privilege level and the prompt specifically scopes this requirement to administrators. On each login, the auth service checks if the admin's `password_set_at` is older than 180 days. If so, it sets `must_change_password = 1` and the middleware redirects them to `/change-password` on every request until they comply. Non-admin roles (organizer, team_lead, member) are not subject to automatic rotation but can be flagged for a forced password change by an administrator at any time for operational reasons (e.g., suspected compromise).
